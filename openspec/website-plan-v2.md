# Arova 網站企劃 v2 — 資訊架構與內容文案規劃

> 基於 PagerDuty、Dynatrace、Resolve、Moveworks、BigPanda 五家競品深度分析

---

## 一、現狀診斷

### 1.1 與競品的差距

| 維度 | Arova 現狀 | 業界標準（競品平均） |
|------|-----------|-------------------|
| **頁面數** | 15 頁 | 50-100+ 頁 |
| **產品頁深度** | 1 頁（平台總覽） | 6-12 頁（每個模組獨立頁面） |
| **解決方案頁** | 5 個方案在 1 頁 | 每方案獨立頁 + 產業頁 + 角色頁 |
| **社會證明** | 3 個匿名案例 | 具名客戶 + logo 牆 + 分析師報告 + 影片 |
| **AI 說明深度** | 「Nexus Intelligence」名稱 + 簡述 | 品牌化技術架構 + 運作流程圖 + 技術元件命名 |
| **整合生態** | 6 分類列表 | 可篩選目錄 + 每個整合獨立頁 + 700+ 數量 |
| **內容行銷** | 6 篇佔位文章 | 100+ 篇 blog + 白皮書 + 影片 + podcast |
| **開發者內容** | 無 | API 文件 + SDK + 社群 + 學習平台 |
| **轉換漏斗** | 只有「預約 Demo」 | 免費試用 + 互動導覽 + ROI 計算器 + gated content |
| **信任中心** | 1 頁概述 | 獨立 trust center + 認證 + 即時狀態 + 滲透測試報告 |

### 1.2 核心問題

1. **產品理解門檻太高**：沒有足夠的頁面深度來解釋 Nexus Intelligence 的運作方式
2. **買家旅程缺失**：只有漏斗底部的 CTA（Demo），沒有頂部和中間的進入點
3. **沒有差異化故事**：缺乏「為什麼選我而不是 ServiceNow / PagerDuty / 自建」的論述
4. **AI 說明太抽象**：「AI-First」宣稱缺乏具體的技術架構和運作機制支撐

## 三、關鍵新增頁面的內容規劃

### 3.1 ★ Nexus Intelligence 專屬頁面 `/platform/nexus-intelligence`

**為什麼需要**：Moveworks 有 Reasoning Engine 專頁、Dynatrace 有 Davis AI 專頁、Resolve 有 RITA/Jarvis 專頁。AI 引擎是 Arova 的核心差異化，必須有深度說明。

**頁面結構**：

```
1. Hero
   標題：「Nexus Intelligence — 為企業 IT 維運設計的 AI 引擎」
   副標：不是通用 AI。不是腳本自動化。而是理解企業 IT 場景、在治理護欄內自主行動的專業 AI。

2. 運作流程（最重要的區塊）
   學習 Moveworks 的 Understand → Plan → Execute → Adapt 四步驟模型
   Arova 版本：
   ┌─────────┐   ┌─────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
   │  感知    │──▶│  理解    │──▶│  規劃    │──▶│  執行    │──▶│  學習    │
   │ Perceive │   │Understand│   │  Plan    │   │ Execute  │   │  Learn   │
   └─────────┘   └─────────┘   └──────────┘   └──────────┘   └──────────┘
   每步驟配：圖示 + 50 字說明 + 具體範例

3. 技術元件（學習 Moveworks 公開內部元件名稱的做法）
   - 意圖解析引擎 (Intent Parser)
   - 上下文記憶體 (Context Memory)
   - 任務規劃器 (Task Planner)
   - 治理驗證器 (Governance Validator)
   - 執行協調器 (Action Orchestrator)
   - 知識累積器 (Knowledge Accumulator)

4. 治理優先的 AI（核心差異化）
   Arova 的獨特角度：不是「AI 做所有事」而是「AI 在企業治理框架內做事」
   - 可配置的自動化邊界
   - 人機協作審批流程
   - 完整的 AI 行為稽核軌跡
   - 可解釋的決策過程

5. vs 傳統方法比較
   表格：Nexus Intelligence vs 規則引擎 vs 通用 LLM vs 自建腳本

6. CTA
   「看看 Nexus Intelligence 如何處理你的場景」→ Demo
   「下載 Nexus Intelligence 技術白皮書」→ Gated content
```

### 3.2 ★ 獨立解決方案頁面（以 AI 驅動 IT 維運為例）

**為什麼需要**：目前 5 個方案擠在同一頁，每個只有 3 段文字。需要展開為獨立深度頁面。

**頁面結構**：

```
1. Hero
   標題：「AI 驅動的日常 IT 維運」
   副標：讓 Nexus Intelligence 自動處理 L1 工單，你的團隊專注於真正重要的事。
   背景指標：「40% L1 工單自動處理」

2. 痛點場景（具象化）
   用 3 個具體日常場景展示：
   - 場景 A：週一早上，50 封密碼重設請求等著處理
   - 場景 B：權限申請流程需要 3 天、跨 4 個系統
   - 場景 C：新人入職 IT 環境準備需要一週

3. Nexus Intelligence 如何解決（流程圖）
   使用者提交請求 → AI 解析意圖 → 驗證權限 → 自動執行 → 記錄稽核 → 通知完成
   每步驟配動畫或圖示

4. 關鍵能力
   - 自然語言工單處理
   - 跨系統自動化執行
   - 治理護欄內的自主操作
   - 知識累積與持續優化

5. 成果指標
   大字號展示：40% 工單自動化 / 85% 使用者滿意度 / 3 分鐘平均處理時間

6. 相關案例（連結到案例頁）

7. CTA
   「預約一場針對你的 L1 工單的專屬 Demo」
   「下載 L1 自動化 ROI 計算表」
```

### 3.3 ★ 產業頁面（以金融業為例）

**為什麼需要**：Dynatrace、PagerDuty、BigPanda 都有產業頁。企業買家搜尋「金融業 IT 自動化」時需要被找到。

**頁面結構**：

```
1. Hero
   標題：「為金融業設計的 AI-First IT 維運」
   副標：滿足金融監管合規、加速部署安全性、降低維運風險

2. 金融業 IT 挑戰
   - 金融監管對稽核的嚴格要求
   - 7×24 服務可用性的壓力
   - 多系統整合的複雜度
   - 合規報表產出的時間成本

3. Arova Nexus 如何回應（對應挑戰）
   - 自動產出合規稽核報告
   - AI 驅動的根因分析縮短 MTTR
   - 標準化部署流程確保變更安全
   - JIT 存取控制滿足最小權限原則

4. 合規認證
   SOC 2 / ISO 27001 / 相關金融法規對應

5. 客戶案例（連結金融業案例）

6. CTA
   「預約金融業專屬 Demo」
```

### 3.4 ★ Blog `/blog`

**為什麼需要**：所有競品都有活躍的 blog。這是 SEO 流量和思想領袖地位的基礎。

**初始分類**：

| 分類 | 目的 | 首批文章建議 |
|------|------|-------------|
| **觀點** | 思想領袖 | 「什麼是 AI-First IT Operations？完整指南」 |

### 3.5 ★ 免費 IT 維運評估 `/assessment`

**為什麼需要**：漏斗中間缺乏低承諾的進入點。Resolve 有 ROI Calculator，BigPanda 有 Self-guided Tour。

**問卷結構**（5-8 題）：

```
Q1: 你的 IT 團隊規模？ (< 20 / 20-50 / 50-200 / 200+)
Q2: 每月處理多少 L1 工單？
Q3: 使用幾種 IT 管理工具？
Q4: 平均 MTTR 是多少？
Q5: 是否有合規稽核需求？
Q6: 偏好的部署模式？

→ 產出個人化評估報告：
  - 你的自動化潛力分數
  - 預估可節省的人力/時間
  - 推薦的 3 個優先場景
  - CTA：「與我們討論你的評估結果」
```

## 四、首頁內容重構

### 4.1 新的敘事流程

學習 Dynatrace 和 Moveworks 的首頁敘事策略，重新設計首頁的區段順序：

```
1. Hero — 量化價值主張 + 視覺化
   「首日自動處理 40% L1 工單，MTTR 縮短 50%」
   ✓ 保留現有

2. ★ Logo 牆 — 「受到這些企業信任」
   即使目前客戶少，也要有這個區塊
   可用「技術合作夥伴」或「ASUS ExpertHub 認證」代替

3. 核心價值（更快/更穩/更可控）
   ✓ 保留現有，但數字要更突出

4. ★ Nexus Intelligence 概覽（取代現有 Platform Overview）
   學習 Moveworks Reasoning Engine 的呈現方式
   感知 → 理解 → 規劃 → 執行 → 學習 的 5 步驟流程圖
   這是首頁最重要的區塊 — 讓訪客立即理解「AI 在這裡怎麼運作」

5. 解決方案預覽
   ✓ 保留現有，但改為卡片連結到獨立方案頁

6. ★ 客戶成果數字牆
   4-6 個大數字：
   「3天→2小時」「MTTR -45%」「L1 工單 -80%」「首日 IT 就緒 100%」
   學習 PagerDuty 和 Moveworks 的 metrics wall

7. 信任證明
   合規認證 + 部署模式 + 企業整合（精簡版，連結到 /trust）

8. ★ 多路徑 CTA（取代單一 CTA）
   - 「預約 Demo」— 漏斗底部
   - 「免費評估 IT 維運自動化潛力」— 漏斗中間
   - 「下載 AI-First IT Ops 指南」— 漏斗頂部
```

### 4.2 Hero 區改善建議

**現狀問題**：tagline 是數字（40% L1、50% MTTR），雖然具體但缺乏情感連結。

**建議改為**：
- 主標題：「讓 IT 團隊從救火模式走向策略模式」（情感）
- 副標題：「Nexus Intelligence 驅動的 AI-First 維運平台，首月即可自動處理 40% L1 工單、縮短 50% 事件回應時間。」（數據佐證）
- 這樣同時打到 CIO（策略層面）和 IT 主管（數字層面）

## 五、內容文案策略

### 5.1 品牌語調指南

| 維度 | 指引 | 範例 |
|------|------|------|
| **專業但不生硬** | 避免學術用語，用商業語言 | ❌「異質系統間的互操作性」→ ✅「讓你的 20 種工具自動協作」 |
| **具體不抽象** | 每個宣稱都要有數字或場景支撐 | ❌「提升維運效率」→ ✅「L1 工單處理從 2 小時縮短至 3 分鐘」 |
| **自信但不誇大** | 用「協助」而非「徹底改變」 | ❌「革命性的 AI 平台」→ ✅「為企業 IT 場景設計的 AI 引擎」 |
| **技術可信** | 適度展示技術深度 | 展示架構元件名稱（Intent Parser、Governance Validator） |
| **雙語一致** | 中英文傳達同等深度 | 不是翻譯，是重新書寫 |

### 5.2 CTA 文案層次化

| 漏斗階段 | CTA 文案 | 出現位置 |
|---------|---------|---------|
| **認知期** | 「下載 AI-First IT Ops 完整指南」 | 資源頁、Blog、首頁 |
| **考慮期** | 「免費評估你的 IT 維運自動化潛力」 | 首頁、方案頁、角色頁 |
| **評估期** | 「預約 30 分鐘平台導覽」 | 平台頁、案例頁 |
| **決策期** | 「取得客製化報價」 | 聯絡頁 |

### 5.3 每頁獨特化 CTA（消除重複）

**問題**：現在 7/15 頁的 CTA 都是「預約 Demo」。

**改善**：

| 頁面 | 新 CTA |
|------|--------|
| 首頁 | 「免費評估你的自動化潛力」 |
| 平台 | 「下載平台架構白皮書」 |
| Nexus Intelligence | 「看 Nexus Intelligence 處理你的場景」 |
| 解決方案（各頁） | 「計算這個場景的 ROI」 |
| 案例 | 「與我們討論類似的場景」 |
| 資源 | 「訂閱 Arova 觀點週報」 |
| 信任 | 「下載安全白皮書」 |
| 角色頁 | 「預約 {角色} 專屬簡報」 |

## 十、總結

### 最關鍵的 3 件事

1. **建立 Nexus Intelligence 專屬頁面** — 這是 Arova 的核心差異化。沒有這個頁面，「AI-First」只是一個口號。學習 Moveworks 的 Reasoning Engine 頁面和 Dynatrace 的 Davis AI 頁面。

2. **增加買家旅程的進入點** — 目前只有「預約 Demo」（漏斗底部）。需要加入免費評估（中間）和可下載內容（頂部），才能捕獲處於不同階段的潛在客戶。

3. **從 1 頁概覽升級為深度子頁面** — 每個解決方案、每個平台模組都需要獨立頁面。這不只是 SEO 需要，更是讓企業買家深入了解產品的必要條件。
